# SUKI CRM — Work Log (25 June 2026)

## Session Focus
Opportunity Detail UI refactor, stage-form integration, and quotation/RFQ workflow wiring in the Sales Pipeline module.

## 1. Opportunity Detail Page Refactor

### 1.1 Removed the Red-Box Requirement Gathering Tab
- Deleted the old `Req. Gathering` tab from `app/(dashboard)/sales-pipeline/[id]/opportunity-detail/page.tsx`.
- Removed the entire third "Tabs" card (Stakeholders, Line Items, RFQs, Quotations, Stage History, Activities) to eliminate duplicate content and blank space.
- The page now has a clean two-box layout: **Opportunity Summary** → **Pipeline Stage Card**.

### 1.2 Stage-Based Form as Main Editable Area
- Created a `renderStageForm()` renderer inside the second box (below the stage stepper).
- The form content changes dynamically based on `deal.status`:
  - **Requirement Gathering** — customer info, business requirements, technical requirements, commercial info, internal notes.
  - **MeetingScheduled / DemoConducted** — combined "Meeting & Demo" form with meeting details and demo details in separate bordered sections.
  - **ProposalSent** — proposal fields plus a new quotation guide panel.
  - **Negotiation** — negotiation fields.
  - **Won / Lost** — read-only summary of all captured details.
- Each stage form uses a wide, clean grid layout (`grid-cols-1 md:grid-cols-2` and `lg:grid-cols-3` for read-only Won/Lost).
- Save button is part of the form and respects `canEditOpportunity()` permissions.

### 1.3 Pipeline Stage Simplification
- Merged `MeetingScheduled` and `DemoConducted` into a single pipeline stage label **"Meeting & Demo"**.
- Removed `DemoConducted` from the active pipeline stepper (`PIPELINE_STAGES` array) in the opportunity detail page.
- Updated `STAGE_LABELS` so `DemoConducted` renders as "Meeting & Demo Details (Historical)" when viewing legacy data.
- Updated `pipeline-list` stage tabs and labels to match: `Meeting` → `Meeting & Demo`, removed `Demo` tab.
- Updated `STAGE_REQUIRED_FIELDS` so moving from Requirement Gathering requires meeting fields; Demo/Negotiation have no required fields to advance.

### 1.4 Stage Advancement Without a Modal
- Removed the separate "Change Stage" modal and replaced stage advancement with an inline **Save & Move** button inside the stage form.
- `handleSaveAndMove(toStage)` saves the current stage details and calls `POST /api/opportunities/{id}/stage-change` in one flow.
- Only enabled when mandatory fields for the current stage are filled.

### 1.5 Reviewing Completed Stages
- The pipeline stepper is now interactive: clicking a completed or active stage sets `viewingStage` and renders that stage's form in review mode.
- Added `handleSaveCompletedStage()` which saves edits with `stage_context` so the audit log records the stage being reviewed.
- A "Back to current stage →" link resets `viewingStage` to `null`.

### 1.6 Stage-Gate Validation Message
- Updated the stage-gate validation message to reference the "stage form" instead of the removed "Req. Gathering tab".

### 1.7 Dead-Code Cleanup
- Removed `TabKey` type, `activeTab` state, `tabs` array, and all tab-related JSX.
- Removed unused icon imports (`Users`, `History`, `Briefcase`, `XCircle`, `Star`, `Plus`, `Clock`, `X`, `formatDateTime`, `Timeline`).
- Removed stage-specific action buttons that only opened tabs (Save Requirements, Update Meeting, Save Demo Notes, Send Proposal, Save Negotiation). Mark Won remains.

## 2. Backend API Updates

### 2.1 Opportunity Details API — `app/api/opportunities/[id]/details/route.ts`
- Added `stage_context` body support so the audit log can record which stage was edited.
- Audit message now reads:
  - `Updated <stageContext> stage details for opportunity "..."` when `stageContext` is provided.
  - `Updated requirement gathering details for opportunity "..."` as fallback.
- No changes to `deal.status`; stage transitions remain the responsibility of the dedicated stage-change endpoint.

### 2.2 New Opportunity Context API — `app/api/opportunities/[id]/context/route.ts`
- Created `GET /api/opportunities/[id]/context` to return compact context needed to pre-fill a new RFQ or Quotation.
- Returns:
  - opportunity id / code / name / value
  - account id / name / code
  - primary contact + full stakeholder list
  - assigned user
  - linked RFQ (latest)
  - `primaryProductId` and `primaryProductName` intentionally `null` (schema does not yet store product of interest).
- Includes auth, tenant scope, and row-level scope checks.

## 3. Quotation / RFQ Workflow Wiring

### 3.1 `ProposalQuotationGuide` Component
- Added a new helper component inside `opportunity-detail/page.tsx`.
- Displays the quotation state for `ProposalSent` and `Negotiation` stages:
  - **No quotation** → prompts to create RFQ or Direct Quotation.
  - **Draft quotation** → shows draft details and "Send to Customer" action.
  - **Sent quotation** → shows sent status and actions: View, Mark as Accepted, Negotiate Changes.
  - **Under negotiation** → shows under-negotiation status with View / Accept actions.
  - **Accepted quotation** → shows acceptance and a "Mark Deal as Won" button.
- Handles async actions: send, accept, negotiate, mark won, with loading states and toast feedback.

### 3.2 Linked Quotations
- Added `linkedQuotations` state and `fetchLinkedQuotations` callback in the opportunity detail page.
- Fetches from `/api/opportunities/${id}/quotations` on page load.
- Passed into `ProposalQuotationGuide` to drive the proposal state UI.

### 3.3 Hero Action Buttons
- Added **Create RFQ** and **Direct Quotation** buttons to the opportunity summary hero card, visible only for `ProposalSent` and `Negotiation` stages.
- Existing **Create Quotation** button label changed to **Direct Quotation** for clarity.
- Added `MessageSquare` icon import for the RFQ button.

## 5. Pipeline-List UI Refactor (All Sub-Modules)

### Goal
Apply the same 2-box layout across every Sales Pipeline sub-module page.

### Files Changed

#### `app/(dashboard)/sales-pipeline/pipeline-list/page.tsx`
- **Removed** the 4 separate KPI `SummaryCard` boxes (Active, Value, Overdue, Won) — their data is now consolidated into the first hero card.
- **Added** a compact hero summary card as the first box:
  - Current stage/filter badge (e.g., "All", "Qualified", "Overdue")
  - Filtered view title (e.g., "All Opportunities", "Meeting & Demo")
  - Active count + pipeline value
  - Overdue warning (on All tab only)
  - Export action button
- **Second box** is now a unified `crm-card` containing:
  - Stage tabs as filters
  - Opportunities table
- **Removed duplicate** Export button from the `PageShell` header.
- Cleaned up unused imports: `SummaryCard`, `Briefcase`, `TrendingUp`, `CheckCircle`, `Clock`, `kpiWon`.

### Why This Covers All Sub-Modules
All sidebar items (All Opportunities, Qualified, Requirement Gathering, Meeting & Demo, Proposal, Negotiation, Overdue, Lost) route to `sales-pipeline/pipeline-list` with different `tab` query parameters. Refactoring the single pipeline-list page therefore applies the 2-box layout to every sub-module automatically.
